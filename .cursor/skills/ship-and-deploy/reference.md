# Ship & Deploy reference

Facts below were verified against repository sources listed in [MANIFEST.json](MANIFEST.json). **Do not edit this file from memory** — re-read sources or run the audit script when they change.

## Git hooks (Husky)

| Hook | Command | When |
| --- | --- | --- |
| `pre-commit` | `npm run lint` | Every commit |
| `pre-push` | `npm run lint` | Every push |

`npm run lint` expands to:

- `npm run lint:frontend` → `apps/frontend` eslint
- `npm run check:migrations` → backend non-destructive migration policy script

Install hooks: `npm install` at repo root (runs `prepare` → husky).

## Root npm scripts (deploy-related)

| Script | Command |
| --- | --- |
| `docker:up` | `docker compose up --build` |
| `docker:up:frontend` | `docker compose up --build frontend` |
| `docker:up:backend` | `docker compose up --build backend` |
| `docker:up:gpu` | `docker compose -f docker-compose.yml -f docker-compose.gpu.yml up --build` |
| `docker:up:obs` | `docker compose -f docker-compose.yml -f docker-compose.observability.yml up --build` |
| `docker:down` | `docker compose down` |
| `docker:down:obs` | compose down with observability override |

**Production-like redeploy (detached, scoped — preferred):**

```bash
# frontend-only
docker compose up --build -d frontend

# backend-only
docker compose up --build -d backend
```

**Full-stack redeploy (exception only):**

```bash
docker compose down && docker compose up --build -d
```

Never append `-v` on production-like hosts.

## Scoped redeploy: path → service map

Use `git diff --name-only <baseline>..HEAD` to pick services:

| Paths | Service(s) to rebuild |
| --- | --- |
| `apps/frontend/**` | `frontend` |
| `apps/backend/**` | `backend` |
| `apps/backend/migrations/**` | `backend` (review migration safety first) |
| `docker-compose.yml` | Usually `frontend` + `backend`; ask if postgres/SeaweedFS service blocks changed |
| `docker-compose.gpu.yml` | `backend` with GPU compose override |
| `docker-compose.observability.yml` | observability services only |
| `docs/**`, `.cursor/**`, `**/*.md` | No deploy |
| `ops/security/**` | Ask user; default no deploy |

**Stateful services — leave running unless explicitly required:**

- `postgres` (DB data in `postgres-data` volume)
- `seaweedfs-*` (object storage in named volumes)
- `model-cache` volume (downloaded ML weights)

## Production data safety checklist

Before any deploy command, confirm:

- [ ] Scope is limited to affected services (not full stack by default)
- [ ] No `-v` / `volume rm` / `prune --volumes`
- [ ] No destructive SQL or credential changes
- [ ] New Alembic migrations reviewed if redeploying `backend`
- [ ] Postgres remains bound to localhost unless user explicitly requested remote exposure

### Forbidden without explicit user approval

| Command / action | Risk |
| --- | --- |
| `docker compose down -v` | Wipes Postgres, SeaweedFS, model cache volumes |
| `docker volume rm …` | Permanent data loss |
| `docker system prune --volumes` | Permanent data loss |
| Full `docker compose down` when scoped `up --build -d <svc>` suffices | Unnecessary downtime |
| Restart `postgres` / `seaweedfs-*` for frontend-only changes | Risk to live data + wasted time |
| `ALTER USER … PASSWORD` on initialized Postgres volume | Can desync app credentials |
| `PDCARE_POSTGRES_PORT_BIND=0.0.0.0:5432` without firewall review | Exposes database |

### Safe patterns

| Scenario | Command |
| --- | --- |
| UI-only fix | `docker compose up --build -d frontend` |
| API-only fix | `docker compose up --build -d backend` |
| Both changed | `docker compose up --build -d frontend backend` |
| Docs-only change | Skip deploy |
| Stop stack (maintenance window) | `npm run docker:down` — preserves volumes, but stops all services |

## TLS / frontend notes

- Frontend container uses `tls-gateway.cjs` with certs from `/etc/letsencrypt` (host mount).
- Default domain env: `LETSENCRYPT_DOMAIN` → `pd.lu.im.ntu.edu.tw`.
- Client API base: `NEXT_PUBLIC_API_BASE_URL` defaults to `/api` in compose build args.

## Commit message patterns (recent)

Examples from repository history:

- `fix(admin-history-overview): default to grouped view on load`
- `feat(admin): add history overview across backend and frontend`
- `refactor(backend): split oversized staff routes and tests`

## Agent test policy (near ship)

From `AGENTS.md`:

- Do not run full test suites during normal implementation.
- Allowed before commit/push when shipping: hook-triggered lint, or explicit user request for tests.

## Volume / data persistence

- `docker compose down` preserves named volumes (Postgres, SeaweedFS, model cache).
- `docker compose down -v` **destroys** persisted data — **never** on production unless the user explicitly requests a wipe.
- Changing `PDCARE_POSTGRES_PASSWORD` in compose does not rotate credentials for an already-initialized Postgres volume (see root `README.md`).

## Known compose warnings

- `version` key in `docker-compose.yml` is obsolete (harmless warning).
- Observability containers may appear as orphans when running base compose only.
